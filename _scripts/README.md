# 자체감사결과 스크래핑 재개 방법

`full_scrape.js`는 공공감사포털(pap.go.kr) 자체감사결과 목록을
2025-07-07 ~ 2026-07-07, 기관분류=공공기관 조건으로 전부 내려받는
스크립트입니다. `checkpoint.json`에 진행 상황(다음 페이지, 파일명
중복 처리 registry)이 저장되어 있어 중단 후에도 이어서 실행할 수
있습니다.

## 재개 방법

1. 이 폴더(`_scripts/`)를 작업 디렉터리로 사용할 임시 경로에 복사
   (checkpoint.json은 스크립트와 같은 폴더에 있어야 함)
2. 실행:
   ```
   node full_scrape.js /home/user/data/자체감사결과
   ```
3. `checkpoint.json`의 `nextPage`부터 자동으로 이어서 진행됨
4. 진행되며 생성되는 파일은 `/home/user/data/자체감사결과/`에 저장됨
5. `progress.log`에 전체 처리 로그가 계속 append됨

## 증분수집 (새로 등록된 보고서만 추가)

`incremental_scrape.js`는 이미 수집한 데이터는 건드리지 않고, **마지막
수집 이후 새로 등록된 보고서만** 추가로 내려받는 스크립트입니다.

```
node incremental_scrape.js /home/user/data/자체감사결과
```

동작 방식:

- **워터마크(watermark)**: 이미 수집한 것 중 가장 최신 등록일(`frstRegDt`)을
  기준선으로 삼습니다. `checkpoint.json`의 `regDtWatermark`에 저장되며,
  없으면 `progress.log`에서 한 번 계산합니다. 포털 API가 최신순으로
  내려주므로 page 0부터 훑다가 워터마크 아래로 내려가면 즉시 멈춥니다
  → **오래된 페이지·파일은 다시 내려받지 않습니다.**
- **경계 처리**: 등록일이 날짜 단위(YYYY-MM-DD)라, 워터마크 당일에
  등록된 보고서는 다시 확인(`>=`)합니다. 이때만 소수의 파일을 다시
  받아 SHA-256 해시로 비교하고, 이미 저장된 내용이면 건너뜁니다.
- **번호 재부여는 해당 그룹만**: 새 문서는 항상 워터마크보다 최신이므로
  자기 그룹의 `(1)`이 되고, 그 그룹의 기존 파일만 한 칸씩 밀려
  이름이 바뀝니다(`(1)→(2)`...). 나머지 그룹은 전혀 건드리지 않습니다.
  → **전체 파일을 훑어 다시 번호 매기지 않습니다.**
- 첨부파일이 없는 보고서는 증분수집에서 건너뜁니다(빈 번호 슬롯을
  만들지 않음).

옵션(환경변수):

- `DRY_RUN=1` : 실제로 저장/이름변경하지 않고 계획만 로그로 출력(미리보기)
- `SEARCH_BGNG` / `SEARCH_END` : 검색 창(YYYYMMDD). 기본 시작=20250707,
  기본 끝=오늘 날짜. 끝을 넓히면 수집 범위가 확장됩니다.

## 주의

- 프록시 포트가 세션마다 바뀔 수 있으므로 `lib.js`가 `$HTTPS_PROXY`
  환경변수에서 자동으로 읽어옴 (하드코딩 아님)
- 검증 규칙: 각 보고서의 모든 조치사항 첨부파일을 실제로 조회해서
  파일명+크기가 다르면 별도 문서로 취급, 같으면 1개로 합침. 파일명은
  `기관명_{연도}년 {감사분야}` 규칙이며 텍스트가 겹치면 등록일
  최신순으로 `(1)`, `(2)`... 번호를 붙임
