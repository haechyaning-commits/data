## 폴더 구조

```
자체감사결과/
  {감사실시기관}_{연도}년 {감사분야}.{확장자}
  {감사실시기관}_{연도}년 {감사분야}(2).{확장자}
  ...
```

- 파일명은 `기관명_연도년 감사분야` 형식입니다 (예: `대한무역투자진흥공사_2026년 종합감사`).
- 감사분야는 API의 `adFldNm` 값(종합감사/특정감사/복무감사/성과감사/재무감사 등)을 사용합니다.
- 같은 기관·연도·감사분야로 여러 건이 등록된 경우, **실제 첨부문서 내용(SHA-256 해시)이 동일하면 번호 없이 1개만 저장**하고, **내용이 다르면 `(1)`, `(2)`... 순서로 번호를 붙여 모두 저장**합니다. 번호는 등록일 기준 최신순으로 매겨집니다.
- 하나의 감사보고서 안에 여러 개의 "처분요구 및 조치사항" 항목이 있는 경우, 각 항목의 첨부파일을 개별적으로 확인하여 실제로 다른 문서인 것만 별도 파일로 저장했습니다 (같은 파일을 공유하는 항목은 중복 저장하지 않음).

## 수집 방식

공공감사포털의 REST API(`/api/fdadPlanRslt`, `/api/files/filelist`, `/api/files/download`)를 직접 호출하여 실제 존재하는 데이터만 수집했습니다. 임의로 생성하거나 추측한 데이터는 없습니다.

수집은 **2단계**로 진행됩니다.

1. **메타데이터 수집(1단계)** — 전체 보고서 목록을 훑어 각 첨부문서의 식별자·파일명·크기를 먼저 확정하고 파일명과 `(1)`, `(2)` 번호를 계산합니다. 전체 목록을 먼저 알아야 이름이 겹치는 건에 번호를 처음부터 정확히 매길 수 있기 때문입니다. (`_scripts/build_manifest.js`, `_scripts/build_manifest_gap.js` → `_scripts/manifest2.json`)
2. **다운로드(2단계)** — 확정된 목록을 배치 단위로 내려받아 저장·커밋·푸시합니다. (`_scripts/orchestrate2.sh` → `_scripts/download_batch.js`)

## 처리 흐름 (API 호출 → 저장)

> **크롤링이 아니라 API 호출입니다.** 사람이 보는 HTML 화면을 파싱한 것이 아니라, 포털이 내부적으로 쓰는 REST API 엔드포인트를 직접 호출해 JSON·파일을 그대로 받아 저장했습니다. 요청은 프록시로 `CONNECT` 터널을 뚫고 `www.pap.go.kr`에 HTTPS로 보냅니다.

`자체감사결과/` 수집 스크립트(`_scripts/full_scrape.js`) 기준, 첨부파일 1개가 저장되기까지의 과정입니다.

```
기관코드표 준비 (GET /api/instCd)
   └─▶ 목록 페이지 조회 (GET /api/fdadPlanRslt, 기간·기관분류를 파라미터로 전달)
         └─▶ 보고서마다 조치사항(subList) 순회
               └─▶ 첨부 메타 조회 (GET /api/files/filelist/{uuid})
                     └─▶ 파일 다운로드 (POST /api/files/download) → SHA-256 내용 해시 계산
                           └─▶ 해시가 같으면 중복 스킵 / 다르면 (1),(2) 번호 부여
                                 └─▶ 디스크 저장 (>90MB면 _조각(N) 분할)
                                       └─▶ 페이지 끝날 때마다 checkpoint.json 저장(재개용)
```

1. **기관코드표 준비** — `GET /api/instCd?...palaw1InstClsfCd=30`을 한 번 호출해 `기관코드 → 기관명` 대응표를 만듭니다. 일부 보고서의 기관명이 `null`로 오는 경우 코드로 실제 이름을 찾아 넣기 위한 대비책입니다.
2. **목록 조회(페이지 단위)** — `GET /api/fdadPlanRslt?searchYmdBgng=...&searchYmdEnd=...&palawInstClsfCd=30&page=N`. **기간·기관분류 조건은 URL 파라미터로 서버에 넘겨** 포털이 조건에 맞는 것만 응답합니다. 응답 JSON의 `totalElements`로 전체 건수를, `_embedded`로 그 페이지의 보고서 배열을 얻습니다.
3. **첨부 실체 확인** — 보고서의 "처분요구 및 조치사항" 목록(`subList`)을 순회하며 각 항목의 UUID로 `GET /api/files/filelist/{uuid}`를 호출해 실제 파일 메타데이터(fileId·fileSn·fileName·fileSize)를 얻습니다. **추측하지 않고 항상 실제 파일을 확인**하며, 병목이 네트워크 왕복이라 항목들을 병렬(bounded)로 처리합니다.
4. **다운로드 + 내용 해시** — `POST /api/files/download` (body: `{fileId, fileSn}`)로 실제 파일 바이너리를 받고, 받은 바이트로 즉시 **SHA-256 해시**를 계산합니다. 이 해시가 "문서 내용의 지문" 역할을 합니다.
5. **중복 판정 + 이름·번호 확정** — 한 보고서 안에서 해시가 같은 첨부는 1개로 합칩니다. 이름 등록부(`nameRegistry`)에 같은 내용이 이미 있으면 건너뛰고, 같은 이름인데 내용이 다르면 `(1)`, `(2)`… 번호를 붙입니다(먼저 저장된 파일도 소급해 `(1)`로 개명). 번호 순서는 API가 준 원래 순서를 따릅니다. 첨부가 없는 보고서는 로그만 남기고 빈 파일을 만들지 않습니다.
6. **디스크 저장** — 확정된 이름 + 원본 확장자로 파일을 씁니다. 90MB(마진)를 넘으면 GitHub 100MB 제한 때문에 `_조각(1)`, `_조각(2)`의 `.part`로 분할 저장합니다.
7. **체크포인트 저장(재개)** — 페이지가 끝날 때마다 다음 페이지 번호·이름 등록부·누적 건수를 `checkpoint.json`에 저장해, 세션이 끊겨도 그 지점부터 이어서 진행합니다.

> `자체감사파일2/`(과거~공백 구간)는 양이 많아(약 17.5GB) 위 과정을 **① 메타데이터를 먼저 전부 확정(`manifest2.json`) → ② 그 목록을 배치로 다운로드·커밋·푸시**하는 2단계로 분리했을 뿐, 핵심 원리(API 호출 → 해시로 중복 판정 → 이름·번호 확정 → 저장)는 동일합니다.

## 수집 성능 / 병렬 처리

이 데이터의 병목은 CPU가 아니라 **API 요청의 네트워크 왕복 대기 시간**입니다. 요청을 보내고 응답을 기다리는 동안 노는 시간을 여러 요청이 겹쳐 소화하도록 동시성을 높여 속도를 개선했으며, 수집을 진행하며 아래 순서로 단계적으로 조정했습니다.

| 단계 | 개선 내용 | 효과 |
|---|---|---|
| 초기 | 순차 처리 | 기준 |
| 다운로드 동시성 상향 | 동시 8 → 20개 병렬 다운로드 | 다운로드 속도 향상 |
| 메타데이터 동시성 상향 | 조회 동시 16 → 40개 | 1단계 속도 향상 |
| **구조 개선(핵심)** | 보고서를 1건씩 순차 처리하던 것을 **페이지 내 100건 전체를 한꺼번에 병렬 요청**하도록 변경 | 페이지당 **약 5분 → 38초 (약 8배)** |

**핵심 포인트**

- **동시성 한도를 늘린 것보다, 놀던 한도를 실제로 쓰게 만든 것이 컸습니다.** 기존에는 `pLimit(40)`으로 40개까지 병렬 허용해도, 코드가 보고서를 1건씩(`for...await`) 처리해 실제로는 평균 2~3개 슬롯만 사용됐습니다. 페이지 내 모든 보고서의 조회 요청을 한 큐에 몰아넣어 40개 슬롯이 항상 꽉 차게 만들었습니다.
- **번호 순서는 그대로 유지**: 요청은 병렬로 보내지만 결과를 이름 레지스트리에 등록하는 것은 API가 준 원래 순서대로 하므로 `(1)`, `(2)` 번호가 뒤바뀌지 않습니다.
- **왜 무한정 늘리지 않는가**: 동시 연결을 과하게 늘리면 포털이 502로 거부하기 시작합니다. 40 정도가 적정선이며, 소량 발생하는 502는 지수 백오프 재시도(1·2·3초)로 자동 복구합니다.
- 현재 값: 메타데이터 조회 동시 40, 다운로드 동시 20.

## 배치 처리(안정성·용량·재개)

다운로드는 **약 400MB 단위 배치**로 "받기 → 저장 → 커밋 → 푸시 → 로컬 사본 삭제"를 반복합니다. 이유는 세 가지입니다.

- **GitHub 업로드 제한**: 처음 약 1.3GB를 한 번에 푸시하려다 HTTP 413(요청 과대)으로 거부당해 400MB로 축소.
- **디스크 관리**: 전체 용량을 로컬에 쌓아둘 수 없어, 푸시한 파일은 바로 삭제해 공간을 재활용.
- **중단 대비(재개 가능)**: 진행기록(`_scripts/done2.log`)을 배치마다 함께 푸시하므로 세션이 끊겨도 이미 받은 것은 건너뛰고 이어서 재개.

포털이 실제 파일을 내주지 않는 경우(HTTP 204)는 재시도해도 받을 수 없으므로, 건너뛰고 `_scripts/unavailable2.log`에 기록합니다(빈 파일을 만들지 않음).
