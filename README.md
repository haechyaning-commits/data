## 다운로드 방법

`main` 브랜치에 `자체감사파일1` ~ `자체감사파일4` 네 폴더가 모두 포함되어 있습니다(합계 약 26GB).

**전체 받기 (ZIP)**
1. 초록색 **Code** 버튼 → **Download ZIP** (네 폴더 전부, 대용량)
2. 압축 해제 후 원하는 폴더 사용

**특정 폴더만 ZIP으로 받기 (예: `자체감사파일1`만)**

GitHub 웹의 *Download ZIP*은 저장소 전체(≈26GB)를 받으므로, 폴더 하나만 원하면 터미널에서 **sparse-checkout으로 그 폴더만 내려받아 압축**하는 것이 가장 빠릅니다:

```bash
# 1) 자체감사파일1 폴더만 받기 (blob 필터 + sparse-checkout: 다른 폴더는 다운로드 안 함)
git clone --filter=blob:none --no-checkout --sparse \
  https://github.com/haechyaning-commits/data.git data
cd data
git sparse-checkout set 자체감사파일1
git checkout main

# 2) 그 폴더를 zip으로 압축 (약 8.33GB → zip 파일 생성)
zip -r 자체감사파일1.zip 자체감사파일1          # Linux/Mac
#  Windows(PowerShell): Compress-Archive -Path 자체감사파일1 -DestinationPath 자체감사파일1.zip
```

- `--filter=blob:none`는 파일 실물을 필요할 때만 받게 하고, `sparse-checkout set 자체감사파일1`은 그 폴더만 워킹트리에 체크아웃하므로 **다른 폴더는 내려받지 않습니다.**
- 다른 폴더를 원하면 이름만 바꾸세요: `git sparse-checkout set 자체감사파일2`(또는 `자체감사파일3`, `자체감사파일4`).
- `git`이 없거나 GUI만 쓰고 싶다면 폴더 URL(`https://github.com/haechyaning-commits/data/tree/main/자체감사파일1`)을 **download-directory.github.io** 같은 서비스에 넣어 폴더 ZIP을 받을 수도 있지만, 파일이 많고 커서(자체감사파일1은 29,285개·8.33GB) 대부분의 웹 서비스는 실패합니다. 위 sparse-checkout 방법을 권장합니다.

(GitHub 웹 화면은 폴더당 1,000개 파일까지만 미리보기로 표시하지만, ZIP 다운로드나 `git clone`/`git pull`에는 전체 파일이 포함됩니다.)

## 수집 방식

공공감사포털의 REST API(`/api/fdadPlanRslt`, `/api/files/filelist`, `/api/files/download`)를 직접 호출하여 실제 존재하는 데이터만 수집했습니다. 임의로 생성하거나 추측한 데이터는 없습니다. 대용량 구간은 **① 메타데이터를 먼저 전부 확정(manifest) → ② 그 목록을 배치로 다운로드·커밋·푸시**하는 2단계로 진행합니다.

## 처리 흐름 (API 호출 → 저장)

> 사람이 보는 HTML 화면을 파싱한 것이 아니라, 포털이 내부적으로 쓰는 REST API 엔드포인트를 직접 호출해 JSON·파일을 그대로 받아 저장했습니다. 요청은 프록시로 `CONNECT` 터널을 뚫고 `www.pap.go.kr`에 HTTPS로 보냅니다.

첨부파일 1개가 저장되기까지의 과정입니다.

```
기관코드표 준비 (GET /api/instCd)
   └─▶ 목록 페이지 조회 (GET /api/fdadPlanRslt, 기간·기관분류를 파라미터로 전달)
         └─▶ 보고서마다 조치사항(subList) 순회
               └─▶ 첨부 메타 조회 (GET /api/files/filelist/{uuid})
                     └─▶ 파일 다운로드 (POST /api/files/download)
                           └─▶ 내용(파일명·크기 또는 해시)으로 중복 판정 / 다르면 (1),(2) 번호 부여
                                 └─▶ 디스크 저장 (>90MB면 _조각(N) 분할)
                                       └─▶ 진행 상태 저장(재개용)
```

1. **기관코드표 준비** — `GET /api/instCd?...palaw1InstClsfCd=30`으로 `기관코드 → 기관명` 대응표를 만듭니다(기관명이 `null`인 보고서 대비).
2. **목록 조회(페이지 단위)** — `GET /api/fdadPlanRslt?searchYmdBgng=...&searchYmdEnd=...&palawInstClsfCd=30&page=N`. **기간·기관분류는 URL 파라미터로 서버에 전달**합니다.
3. **첨부 실체 확인** — 보고서의 "처분요구 및 조치사항"(`subList`)을 순회하며 UUID로 `GET /api/files/filelist/{uuid}`를 호출해 실제 파일 메타(fileId·fileSn·fileName·fileSize)를 얻습니다. 병목이 네트워크 왕복이라 항목들을 병렬(bounded)로 처리합니다.
4. **다운로드** — `POST /api/files/download` (body: `{fileId, fileSn}`)로 실제 파일 바이너리를 받습니다.
5. **중복 판정 + 이름·번호 확정** — 같은 이름 그룹에서 내용이 같으면 1개만, 다르면 `(1)`, `(2)`… 번호를 등록일 최신순으로 붙입니다. 첨부가 없는 보고서는 빈 파일을 만들지 않습니다.
6. **디스크 저장 / 재개** — 확정된 이름으로 저장(>90MB면 분할)하고, 진행 상태를 저장해 세션이 끊겨도 이어서 진행합니다.

## 수집 성능 / 병렬 처리

병목은 CPU가 아니라 **API 요청의 네트워크 왕복 대기**입니다. 동시성을 높여 대기 시간을 겹쳐 소화하도록 개선했습니다.

- **핵심 개선**: 보고서를 1건씩 순차 처리하던 것을 **페이지 내 100건 전체를 한꺼번에 병렬 요청**하도록 변경 → 페이지당 약 5분 → 38초(약 8배). 동시성 한도를 늘린 것보다 놀던 한도를 실제로 쓰게 만든 것이 컸습니다.
- **번호 순서 유지**: 요청은 병렬이지만 이름 레지스트리 등록은 API가 준 원래 순서대로 하므로 `(1)`, `(2)` 번호가 뒤바뀌지 않습니다.
- **과부하 방지**: 동시 연결을 과하게 늘리면 포털이 502로 거부하므로 적정선을 유지하고, 소량의 502·일시 오류는 지수 백오프(1·2·3초)로 자동 재시도합니다.
- 현재 값: 메타데이터 조회 동시 40, 다운로드 동시 20.

## 배치 처리(안정성·용량·재개)

다운로드는 **약 350~400MB 단위 배치**로 "받기 → 저장 → 커밋 → 푸시"를 반복합니다.

- **GitHub 업로드 제한**: 한 번에 큰 팩을 푸시하면 HTTP 413(요청 과대)으로 거부되므로 배치 크기를 제한합니다.
- **중단 대비(재개 가능)**: 진행기록(done 로그)을 남겨 세션이 끊겨도 이미 받은 것은 건너뛰고 이어서 재개합니다.
- 포털이 실제 파일을 내주지 않는 경우(HTTP 204)는 재시도해도 받을 수 없으므로 건너뛰고 별도 로그에 기록합니다(빈 파일을 만들지 않음).
