# 자체감사결과 데이터 수집 정리

공공감사포털([pap.go.kr](https://pap.go.kr))의 자체감사결과 공개데이터를 수집한 두 개의 데이터셋 정리입니다. 둘 다 **기관분류=공공기관** 조건으로, 포털 REST API(`/api/fdadPlanRslt`, `/api/files/filelist`, `/api/files/download`)를 직접 호출해 실제 존재하는 파일만 저장했습니다. 임의로 생성하거나 추측한 데이터는 없습니다.

**두 데이터셋을 합치면 2021-01-01 ~ 2026-07-07 전 구간이 끊김 없이 연속으로 수집되어 있습니다.**

| 데이터셋 | 수집 기간 | 보고서 | 첨부문서 |
|---|---|---|---|
| 자체감사파일 (`자체감사결과/`) | **2025-07-07 ~ 2026-07-07** | 2,384건 | 10,828개 |
| 자체감사파일2 (`자체감사파일2/`) | **2021-01-01 ~ 2025-07-06** | 13,476건 | 45,853개 |

---

## 1. 자체감사파일 — `자체감사결과/` 폴더 (최신 구간)

| 항목 | 내용 |
|---|---|
| **수집 기간** | **2025-07-07 ~ 2026-07-07** |
| 기관분류 | 공공기관 (`palawInstClsfCd=30`) |
| 규모 | 보고서 2,384건 / 첨부파일 10,828개 (실제 파일 10,830개, 1건은 GitHub 100MB 초과로 2조각 분할) |
| 정렬 | 최신순 (API 기본 정렬) |

**수집 방법**
- 포털 REST API를 직접 호출하는 단일 스크립트(`_scripts/full_scrape.js`)로 목록 조회 → 첨부 다운로드를 진행.
- 파일명 규칙: `{감사실시기관}_{연도}년 {감사분야}.{확장자}` (감사분야는 API의 `adFldNm` 값 — 종합감사/특정감사/복무감사/성과감사/재무감사 등).
- 같은 기관·연도·분야가 여러 건이면 → 첨부문서 **SHA-256 해시가 같으면 번호 없이 1개만**, 다르면 `(1)`, `(2)`… 순으로 번호(등록일 최신순)를 붙여 모두 저장.
- 하나의 보고서에 "처분요구 및 조치사항" 항목이 여러 개면 각 항목 첨부를 개별 확인해 실제로 다른 문서만 별도 저장.
- `checkpoint.json`으로 중단 후 이어받기 지원.

## 2. 자체감사파일2 — `자체감사파일2/` 폴더 (과거 ~ 공백 구간)

두 번에 걸쳐 수집한 데이터를 하나의 폴더에 모은 것입니다.

| 항목 | 내용 |
|---|---|
| **수집 기간** | **2021-01-01 ~ 2025-07-06** (자체감사파일 이전 구간 전체) |
| ├ 1차 | 2021-01-01 ~ 2023-12-30 → 보고서 9,125건 / 첨부 22,406개 |
| └ 2차(공백기간) | 2023-12-31 ~ 2025-07-06 → 보고서 4,351건 / 첨부 23,453개 (1차 번호에 이어서 부여) |
| 합계 | 첨부문서 **45,853개 (약 17.5GB)** |
| 기관분류 | 공공기관 (`palawInstClsfCd=30`) |
| 미수집 | 6건은 포털이 실제 파일을 내주지 않아(HTTP 204) 저장 못 함 → `_scripts/unavailable2.log`에 기록 (빈 파일은 만들지 않음) |

**수집 방법 — 2단계 방식**
1. **메타데이터 수집(1단계)** — 전체 목록을 먼저 훑어 각 첨부의 식별자·파일명·크기를 확정하고 `(1)`, `(2)` 번호를 미리 계산. 전체 목록을 알아야 이름이 겹치는 건에 번호를 처음부터 정확히 매길 수 있기 때문. (`_scripts/build_manifest.js`, `_scripts/build_manifest_gap.js` → `_scripts/manifest2.json`)
2. **다운로드(2단계)** — 확정된 목록을 약 400MB 단위 배치로 "받기 → 저장 → 커밋 → 푸시 → 로컬 사본 삭제" 반복. (`_scripts/orchestrate2.sh` → `_scripts/download_batch.js`, 진행기록 `_scripts/done2.log`로 재개 가능)

- 파일명 규칙·중복 처리(파일명+크기 동일 시 1개)는 `자체감사결과/`와 동일.
- 대용량 3건(`한국도로공사_2022년 특정감사(7).hwp`, `한국농어촌공사_2025년 특정감사(16).hwpx`, `한국농어촌공사_2025년 특정감사(17).hwpx`)은 100MB 초과로 `_조각(1)`, `_조각(2)` `.part` 분할 저장.

**성능 / 병렬 처리**
- 병목은 CPU가 아니라 API 요청의 네트워크 왕복 대기 시간. 동시성을 높이고, 무엇보다 **페이지 내 100건 전체를 한꺼번에 병렬 요청**하도록 구조를 바꿔 페이지당 약 5분 → 38초(약 8배)로 단축.
- 현재 값: 메타데이터 조회 동시 40, 다운로드 동시 20. 과하게 늘리면 포털이 502로 거부하므로 40이 적정선이며, 소량 502는 지수 백오프(1·2·3초)로 재시도 복구.
- 번호 순서는 결과를 API가 준 원래 순서대로 등록해 `(1)`, `(2)`가 뒤바뀌지 않음.

## 대용량 분할 파일 복원

100MB를 초과한 파일은 `_조각(1)`, `_조각(2)`의 `.part`로 분할되어 있습니다. 원본 복원:

**Windows (명령 프롬프트)**
```
copy /b "파일명_조각(1).확장자.part"+"파일명_조각(2).확장자.part" "파일명.확장자"
```

**Mac / Linux**
```
cat "파일명_조각(1).확장자.part" "파일명_조각(2).확장자.part" > "파일명.확장자"
```
